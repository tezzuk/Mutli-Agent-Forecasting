2. The Problem: Why Not Just Average?

The simplest ensemble strategy is to average all agent predictions equally: ŷ = (p_1 + p_2 + p_3 + p_4) / 4. This is the EqualWeightAggregator. It has three fatal weaknesses in financial forecasting:

🚫 Ignores Performance Differences

If TrendAgent consistently predicts well and VolatilityAgent consistently predicts poorly, equal weighting gives them the same influence. The bad agent actively drags down ensemble accuracy.

🚫 Can't Adapt to Regime Changes

In calm markets (VIX=12), MomentumAgent may dominate. In a crash (VIX=80), VolatilityAgent may be the only useful predictor. Equal weighting can't switch emphasis — it applies the same mix regardless of market conditions.

🚫 No Accountability

When the ensemble is wrong, equal weighting assigns equal blame. There's no mechanism to identify which agent caused the error and reduce its future influence.

✅ What We Need Instead

An algorithm that continuously adjusts agent weights based on realized performance, adapts to changing regimes, and has provable theoretical guarantees on how much worse it can be than the best agent.

The solution is online learning — a framework designed precisely for settings where data arrives sequentially and the environment changes over time.

3. The Online Learning Framework

Online learning is a paradigm where a learner makes decisions sequentially, observes outcomes, and updates its strategy — one step at a time, with no access to future data.

The loop is: predict → observe → update → predict → observe → update → ...

Contrast with Batch Learning

Why Financial Forecasting Is Inherently Online

  • Regimes change: The bull market of 2010-2019, the COVID crash of 2020, the rate-hike cycle of 2022 — each required different models to perform well. No single model trained on historical data dominates all of them.
  • Data streams in: Every trading day produces a new data point. You never have "all the data" — new information arrives forever.
  • The environment reacts: When many traders use the same model, its alpha decays. Market structure is non-stationary by design.

The Hedge algorithm is designed for exactly this setting. It doesn't assume any agent is good — it just tracks their performance and weights them accordingly, with a mathematical guarantee that it can't do much worse than the best one.

4. The Hedge Algorithm — Full Mathematical Treatment

4.1 The Setup

We have N agents (N=4 in our project) and run for T time steps (T≈3520 trading days). At each step t:

  • Each agent i produces a prediction pi(t)
  • The ensemble produces ŷt = Σ wi(t) · pi(t)
  • The actual return yt is revealed
  • Weights are updated for the next step

4.2 The Full Algorithm — Step by Step

0Initialize: wi(1) = 1/N for all i = 1, ..., N
Uniform prior — no agent is privileged at the start. This reflects maximum uncertainty.
1Aggregate: ŷt = Σi wi(t) · pi(t)
Weighted average. Agents with higher weights contribute more to the ensemble prediction.
2Observe: yt (actual log return for day t)
3Compute losses: Li(t) = (pi(t) − yt)²
Squared error. Always non-negative. Penalizes large errors quadratically.
4Update: wi(t+1) ← wi(t) · exp(−η · Li(t))
Multiplicative penalty. Agents with large loss get their weight multiplied by a number close to 0. Agents with small loss get weight close to unchanged.
5Normalize: wi(t+1) ← wi(t+1) / Σj wj(t+1)
Project back onto the probability simplex so weights sum to 1.

4.3 Why exp(−η · L)?

The exponential function is the key design choice. Consider two agents: Agent A has loss L=0.0001 (very accurate) and Agent B has loss L=0.01 (10× worse).

After one update with η=0.1:

  • Agent A's weight multiplied by: exp(−0.1 × 0.0001) = exp(−0.00001) ≈ 0.99999 (almost unchanged)
  • Agent B's weight multiplied by: exp(−0.1 × 0.01) = exp(−0.001) ≈ 0.999 (slightly reduced)

But if Agent B has a catastrophic prediction with L=1.0:

  • Agent B's weight multiplied by: exp(−0.1 × 1.0) = exp(−0.1) ≈ 0.905 — a 9.5% reduction in one step

The exponential penalty is proportional to the error magnitude. Small errors → tiny weight reduction. Large errors → significant weight reduction. This is smooth, differentiable, and never reduces a weight to exactly zero (so every agent can recover).

4.4 Why Normalize?

After the multiplicative update, the weights no longer sum to 1. But aggregate() computes a weighted average — which requires the weights to form a probability distribution. Normalizing by the sum restores this property. Geometrically, this projects the weight vector back onto the probability simplex: the set {w : Σ wi = 1, wi ≥ 0}.

💡 Why is ηopt ≈ 0.0094 so small? Financial log returns have very high noise-to-signal ratio. Daily SPY returns have std ≈ 1%, but the predictable component is far smaller (maybe 0.01-0.05%). With so much noise, a single bad day for an agent doesn't mean it's a bad agent — it just had bad luck. A small η prevents over-reacting to noise. The optimal η encodes this: the more time steps (larger T), the smaller the optimal η, because you have more data to distinguish signal from noise.

5. The Regret Bound — Why This Matters

5.1 Definitions

Cumulative loss of ensemble: Lensemble = Σt=1T (ŷt − yt)²

Cumulative loss of best agent: L* = mini Σt=1T (pi(t) − yt)²

Regret: RT = Lensemble − L*

Regret measures how much worse the ensemble performs compared to the best single agent — but evaluated in hindsight, after seeing all T outcomes. In hindsight, you know which agent was best. The Hedge algorithm competes with this oracle.

5.2 The Theorem

Theorem (Freund & Schapire, 1997): For the Hedge algorithm with η = √(log(N) / (2T)):

RT ≤ √(T · log(N) / 2)

5.3 Intuition for the √T Bound

Why √T and not T? This is the key insight of the result.

  • If regret grew as T: The ensemble would be a constant fraction worse than the best agent — no matter how long you run it, it never "catches up." This would mean online learning is useless.
  • Regret grows as √T: The average regret per step is R_T / T ≤ √(log(N) / (2T)) → 0 as T → ∞. The ensemble's per-step performance converges to the best agent's performance. Online learning works.

6. Code Walkthrough — HedgeAggregator

class HedgeAggregator:
    def __init__(self, n_agents: int, eta: float = 0.1):
        self.n_agents = n_agents
        self.eta = eta

        # np.ones(N) / N → array([0.25, 0.25, 0.25, 0.25]) for N=4
        # Uniform prior: every agent starts with equal credibility
        self.weights = np.ones(n_agents) / n_agents

        # weight_history[t] = weight vector after t-th update
        # This is the data powering the "agent weight evolution" chart in the dashboard
        self.weight_history: list[np.ndarray] = []
        self.loss_history:   list[np.ndarray] = []

    def aggregate(self, predictions: list[float]) -> float:
        # np.dot(weights, predictions) = Σ w_i * p_i
        # Single numpy operation — fast even for many agents
        return float(np.dot(self.weights, predictions))

    def update(self, predictions: list[float], actual: float) -> None:
        preds = np.array(predictions, dtype=float)

        # Step 3: squared loss per agent — (p_i - y_t)^2
        # Why squared? Differentiable everywhere, always ≥ 0, penalizes large errors more
        losses = (preds - actual) ** 2

        # Step 4: multiplicative update — w_i *= exp(-eta * L_i)
        # exp(-eta * 0) = 1.0     → accurate agent: weight unchanged
        # exp(-eta * big) → 0   → terrible agent: weight crushed
        self.weights *= np.exp(-self.eta * losses)

        # Step 5: normalize so weights sum to 1 (stay on probability simplex)
        self.weights /= self.weights.sum()

        # Record for later analysis / dashboard visualization
        self.weight_history.append(self.weights.copy())  # .copy() is critical!
        self.loss_history.append(losses.copy())
⚠ Why .copy() is critical self.weights is mutated in-place at every step. Without .copy(), weight_history would store references to the same array — and every historical entry would show the current weights. .copy() creates a new independent array, capturing the state at that exact moment. 
    @staticmethod
    def optimal_eta(T: int, N: int) -> float:
        # Minimizes the regret bound R_T ≤ sqrt(T * log(N) / 2)
        # Derived by taking dR/dη = 0 — balances exploration vs. exploitation
        return np.sqrt(np.log(N) / (2 * T))

    def weight_dataframe(self, index=None, agent_names=None) -> pd.DataFrame:
        # Converts list of arrays → labeled DataFrame for Streamlit charting
        cols = agent_names or [f"agent_{i}" for i in range(self.n_agents)]
        df = pd.DataFrame(self.weight_history, columns=cols)
        if index is not None:
            df.index = index[:len(df)]
        return df

    def reset(self) -> None:
        # Walk-forward backtesting: each fold starts fresh
        # Without reset, weights trained on 2010-2013 carry over to the 2013-2016 fold
        # This would violate temporal causality in the backtest
        self.weights = np.ones(self.n_agents) / self.n_agents
        self.weight_history.clear()
        self.loss_history.clear()

7. EqualWeightAggregator — The Scientific Baseline

class EqualWeightAggregator:
    def __init__(self, n_agents: int):
        self.n_agents = n_agents
        self.weights = np.ones(n_agents) / n_agents

    def aggregate(self, predictions: list[float]) -> float:
        return float(np.mean(predictions))   # identical to np.dot(uniform_weights, preds)

    def update(self, predictions: list[float], actual: float) -> None:
        pass   # intentionally empty — this is the point

    @property
    def weight_history(self) -> list:
        return []   # weights never change, nothing to record

Why is this the right baseline? To claim that the Hedge algorithm adds value, you need to compare it to the simplest possible ensemble. Equal weighting is that baseline — it uses all the agents but makes no attempt to learn which are better.

  • If Hedge ≈ EqualWeight in performance: all agents are roughly equal. Adaptive weighting adds no value. The agents may not be specialized enough.
  • If Hedge >> EqualWeight: regime-specific specialization matters. The TrendAgent really does do better in trending markets, and Hedge correctly discovers this.

The identical interface (both have aggregate() and update()) means you can swap them in the backtest engine with a single line change — enabling clean A/B comparison.

9. η Sensitivity Analysis

η is the single most important hyperparameter in the Hedge algorithm. Here's a concrete analysis for our project:

η = 0.001 (Very Conservative)

After 3520 steps with a typical loss of ~0.0001: weight update factor = exp(−0.001 × 0.0001) ≈ 0.9999999. Weights barely move from 0.25. You get almost the same result as EqualWeight. The algorithm is too slow to adapt.

η = 0.0094 (Optimal)

Theoretically optimal for T=3520, N=4. Provides the regret bound guarantee. In practice, weights shift gradually over months, tracking genuine performance differences rather than day-to-day noise.

η = 0.1 (Default)

More aggressive than optimal. After one terrible prediction (loss=1.0): weight factor = exp(−0.1) ≈ 0.905. An agent can lose 9.5% of its weight in one day. Works well but may overreact to temporary bad spells.

η = 1.0 (Aggressive)

After one bad day (loss=0.01): factor = exp(−0.01) ≈ 0.99 — modest. But after a crash-day prediction (loss=0.25): factor = exp(−0.25) ≈ 0.78 — loses 22% in one step. Near "winner-take-all" dynamics.
📉 Practical Recommendation Start with η=0.1 for initial testing (fast feedback loop). Before final deployment, run a grid search over η ∈ {0.001, 0.005, 0.01, 0.05, 0.1, 0.5} on the walk-forward backtest and pick the η that maximizes Sharpe ratio. The theoretical optimal (η≈0.0094) is a good starting point for the grid search.

10. Full Interview Q&A

Q1. What is the Hedge algorithm and who invented it?
The Hedge algorithm is an online learning method for combining expert predictions with provable performance guarantees. It was introduced by Freund and Schapire in 1997 in their paper "A Decision-Theoretic Generalization of On-Line Learning and an Application to Boosting." The algorithm maintains a probability distribution over N experts (agents), produces a weighted prediction, and updates weights multiplicatively based on each expert's realized loss. The key guarantee: the algorithm's cumulative loss is within O(√(T log N)) of the best single expert in hindsight.
Q2. What is regret in online learning? What does "no-regret" mean?
Regret R_T is the difference between the algorithm's cumulative loss and the best fixed strategy's cumulative loss over T rounds: R_T = L_algorithm − L*. It measures how much you "regret" not having chosen the best strategy from the start. A "no-regret" algorithm is one where average regret per step goes to zero as T → ∞: R_T / T → 0. The Hedge algorithm achieves this — its regret bound is O(√T), so average regret O(1/√T) → 0. This means in the long run, it performs as well as the best agent — even without knowing in advance which agent that is.
Q3. Why does the regret bound grow as √T and not T?
If regret grew as T, the algorithm would be a constant fraction worse than the best agent forever — online learning would be useless. The √T growth comes from the structure of the multiplicative weights update. Intuitively, at each step the algorithm must explore (keep some weight on suboptimal agents to detect regime changes) while exploiting (concentrate weight on the current best agent). The optimal balance — encoded in η = √(log N / 2T) — produces exactly √T growth. The log N term captures the "search space" — with more agents, there's more uncertainty, but it grows only logarithmically.
Q4. Why do you use squared loss (p_i − y_t)² instead of absolute error |p_i − y_t|?
Three reasons. First, squared loss is differentiable everywhere — absolute error has a non-differentiable kink at zero, which complicates theoretical analysis. Second, squared loss penalizes large errors quadratically — a prediction that is 2× worse gets 4× the penalty — which is appropriate for financial returns where large errors are disproportionately damaging. Third, the Hedge algorithm's regret bound proof relies on the squared loss being bounded, which holds naturally for financial log returns (which are bounded in practice, even if not in theory).
Q5. Why normalize the weights after each update?
The aggregate() function computes a weighted average, which requires the weights to form a valid probability distribution (non-negative, sum to 1). After the multiplicative update w_i *= exp(−η·L_i), each weight is multiplied by a number in (0, 1], so the weights remain non-negative but no longer sum to 1. Dividing by the sum restores the sum-to-1 property. Geometrically, this "projects" the weight vector back onto the probability simplex — the set of all valid probability distributions over N agents. Without normalization, aggregate() would compute a weighted sum (not average), and the scale of predictions would drift over time.
Q6. How do you choose η? What happens if it's too large or too small?
The theoretically optimal η = √(log(N) / (2T)), which minimizes the regret bound. For our project (T=3520, N=4), this gives η≈0.0094. Too large (η=1.0): the algorithm over-reacts to noise — a single bad day for an agent causes a large weight reduction even if the agent is genuinely good. This leads to instability. Too small (η=0.001): the algorithm under-reacts — weights barely deviate from uniform no matter how different the agents perform. In practice, I use η=0.1 as the default (validated empirically) and recommend tuning it via walk-forward cross-validation over the grid {0.001, 0.01, 0.1, 0.5}.
Q7. What is the difference between online learning and batch learning? Why is online learning appropriate here?
In batch learning, you train a model on a fixed dataset and deploy it unchanged. This assumes the data is i.i.d. from a fixed distribution — valid for image classification, but not for financial markets. Financial data is non-stationary: the 2010-2019 bull market, the 2020 COVID crash, and the 2022 rate-hike cycle required completely different models to predict well. Online learning makes no stationarity assumption — it works even if an adversary is choosing the data sequence. It adapts continuously: every day's outcome updates the agent weights. The Hedge algorithm specifically is designed for the "prediction with expert advice" setting, which exactly matches our problem: multiple agents (experts) provide predictions, we combine them, and we observe the outcome.
Q8. Why does reset() need to be called between backtest folds?
Walk-forward backtesting divides the data into expanding training windows: train 2010-2012 / test Q1 2013, then train 2010-2013 / test Q2 2013, etc. If we don't reset the Hedge weights between folds, the weights entering fold k reflect performance from folds 1 through k-1. This means the weight update in fold k implicitly uses information from future data relative to the training period being evaluated. For example, if VolatilityAgent happened to have a great 2011-2012, it might enter Q3 2013 with high weight — but that's a coincidence that wouldn't be known in real-time. Resetting to uniform at each fold correctly simulates what the algorithm would have known at the start of that test period.
Q9. If one agent makes a terrible prediction early on, can it ever recover its weight?
Yes — this is one of the key properties of the multiplicative weights update. The update w_i *= exp(−η·L_i) multiplies the weight by a positive number, never by zero. So no agent's weight ever reaches exactly zero. If the agent subsequently makes many accurate predictions (L_i ≈ 0), its weight factor is exp(0) = 1 each step — weight unchanged. Meanwhile, if other agents make errors, their weights decrease. Through this relative mechanism, a previously-penalized agent can gradually recover as long as it outperforms others in subsequent steps. This is the "exploration" benefit: maintaining some probability on every agent ensures no specialist is permanently silenced before it gets a chance to prove itself in a new regime.
Q10. How does Hedge relate to Boosting / AdaBoost?
Both Hedge and AdaBoost are instances of the Multiplicative Weights Update (MWU) framework. The key difference is direction: AdaBoost reweights training examples to focus on the hard-to-classify ones (iterative model training), while Hedge reweights models/experts based on their prediction performance (online ensemble aggregation). Freund and Schapire showed in the same 1997 paper that AdaBoost can be viewed as a special case of Hedge applied to the training set. Both share the same theoretical foundation — exponential weight updates with regret/generalization bounds — but serve different purposes: AdaBoost is a batch algorithm for training stronger classifiers, Hedge is an online algorithm for combining pre-trained forecasters.
Q11. What is the probability simplex and why do we project back onto it?
The probability simplex Δ_N is the set of all valid probability distributions over N items: Δ_N = {w ∈ ℝ^N : w_i ≥ 0 for all i, and Σ w_i = 1}. For N=4, it's a tetrahedron in 4D space. The weighted average aggregate() = Σ w_i·p_i only has a clean interpretation when w lies on Δ_N — otherwise it's a weighted sum with an arbitrary scale. After the multiplicative update, weights remain non-negative (exp is always positive) but don't sum to 1. Dividing by the sum Σ w_j projects them back onto Δ_N — this is a simple but exact projection. More generally, the Hedge algorithm can be understood as mirror descent on the simplex with an entropic regularizer (negative entropy), and the normalization step is the projection in that mirror descent framework.
Q12. Why store weight_history as a list of arrays rather than a 2D array?
A list of arrays grows dynamically — you append one row at each update step without knowing T in advance. A pre-allocated 2D array would require knowing T before starting, which isn't the case in our walk-forward backtest where T grows fold by fold. The list approach is O(1) amortized per append. When you need to analyze the history (for the dashboard), weight_dataframe() converts it to a DataFrame in one shot: pd.DataFrame(self.weight_history) stacks the list of 1D arrays into a 2D matrix efficiently. This pattern — dynamic list for streaming, then batch conversion for analysis — is idiomatic pandas/numpy practice.
