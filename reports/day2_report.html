2. Architecture — Why a Uniform Interface?

The Hedge aggregator (Day 3) needs to call each agent's prediction at every time step and update their weights based on realized loss. For this to work, every agent must look identical to the aggregator — same input, same output, same method names. This is enforced via an Abstract Base Class.

2.1 BaseAgent — The Contract

from abc import ABC, abstractmethod

class BaseAgent(ABC):
    """
    ABC = Abstract Base Class.
    Python will REFUSE to instantiate any subclass that doesn't implement
    every @abstractmethod. This is a compile-time contract, not a runtime check.
    """

    @abstractmethod
    def fit(self, train_df: pd.DataFrame) -> None: ...
    # train_df: the full feature matrix for the training period
    # Must include 'log_return' column (the target)

    @abstractmethod
    def predict(self, test_df: pd.DataFrame) -> np.ndarray: ...
    # Returns a 1-D NumPy array of predicted log returns, one per row

    @property
    def name(self) -> str:
        return self.__class__.__name__
    # e.g. "TrendAgent", "MomentumAgent" — used for labelling dashboard plots

What ABC enforces

If a subclass forgets to implement fit() or predict(), Python raises TypeError: Can't instantiate abstract class ... with abstract methods fit, predict — at instantiation time, not at first call. Bugs are caught early.

Why @abstractmethod and not just raise NotImplementedError?

raise NotImplementedError is a runtime check — you only discover the bug when you call the method. @abstractmethod is a structural check — you can never even create the object if the contract is broken.

Why pass the full DataFrame?

Each agent needs different columns. TrendAgent uses only the date index. MomentumAgent uses lag + RSI + MACD. Passing the full DataFrame and letting each agent select its columns internally keeps the interface clean and uniform.

Why the name property?

The Streamlit dashboard plots agent weight evolution over time. Each line needs a label. agent.name gives "TrendAgent", "MomentumAgent" etc. automatically — no manual string management.

3. Agent 1 — TrendAgent

What it captures: The long-run directional drift of the S&P 500 (markets trend upward over decades) and annual seasonal patterns (January effect, summer doldrums, year-end rally). Uses only the date index — no price or return features.

3.1 Full Code + Explanation

from statsmodels.tsa.deterministic import CalendarFourier, DeterministicProcess

class TrendAgent(BaseAgent):
    def __init__(self, fourier_order: int = 4):
        self.fourier_order = fourier_order
        self._model = LinearRegression()
        self._dp = None

    def _make_dp(self, index: pd.DatetimeIndex):
        # CalendarFourier: generates sin/cos pairs for annual seasonality
        # order=4 means 4 harmonics → 8 columns (sin1,cos1, sin2,cos2, sin3,cos3, sin4,cos4)
        fourier = CalendarFourier(freq="Y", order=self.fourier_order)

        # DeterministicProcess builds the full design matrix:
        #   constant=True → intercept column (β₀)
        #   order=1       → linear time trend column t (β₁·t)
        #   additional_terms=[fourier] → 8 sin/cos columns
        #   drop=True → removes any perfectly collinear columns
        return DeterministicProcess(
            index=index, constant=True, order=1,
            additional_terms=[fourier], drop=True,
        )

    def fit(self, train_df: pd.DataFrame) -> None:
        self._dp = self._make_dp(train_df.index)
        X = self._dp.in_sample()     # design matrix for training dates
        y = train_df["log_return"].values
        self._model.fit(X, y)

    def predict(self, test_df: pd.DataFrame) -> np.ndarray:
        # out_of_sample: generates the same design matrix for NEW dates
        # steps=len(test_df), index=test_df.index → one row per test date
        X = self._dp.out_of_sample(steps=len(test_df), index=test_df.index)
        return self._model.predict(X)

3.2 What is DeterministicProcess?

DeterministicProcess from statsmodels constructs a design matrix of deterministic (non-random) basis functions of time. For a given date index, it produces columns like:

X = [1, t, sin(2πt/365), cos(2πt/365), sin(4πt/365), cos(4πt/365), ...]

where t = integer day number since the start of the series

This is then fitted with OLS (LinearRegression): log_return ≈ β₀ + β₁·t + β₂·sin(2πt/365) + β₃·cos(2πt/365) + ...

3.3 What is CalendarFourier?

Fourier terms model periodic (cyclical) patterns by decomposing them into sine and cosine waves at different frequencies. For freq="Y", order=4:

HarmonicColumns GeneratedPeriod Captured
1st (fundamental)sin(2πt/365), cos(2πt/365)Annual cycle (Jan–Dec)
2ndsin(4πt/365), cos(4πt/365)Semi-annual (2× per year)
3rdsin(6πt/365), cos(6πt/365)4-month cycle
4thsin(8πt/365), cos(8πt/365)Quarterly (4× per year)

Higher harmonics capture finer seasonal patterns. Order=4 gives 8 Fourier columns — enough to represent complex annual seasonality without overfitting.

3.4 in_sample() vs out_of_sample()

  • in_sample() — generates the design matrix for the dates the process was built on (training dates). Used in fit().
  • out_of_sample(steps, index) — generates the design matrix for new dates not seen during training. Used in predict(). The sin/cos formulas are just evaluated at the new date values — no data needed.
💡 Why no price features for TrendAgent? The whole point of TrendAgent is to capture the structural drift of markets — the component that would exist even without any short-term noise. Price features (RSI, MACD, lags) add short-term noise that would drown out the long-run signal. By using only the date index, TrendAgent produces a smooth baseline prediction that the other agents then deviate from.

4. Agent 2 — MomentumAgent

What it captures: Short-term serial dependence and momentum patterns — the tendency for recent returns to predict near-future returns. XGBoost excels here because momentum effects are non-linear (they appear in certain regimes and disappear in others) and interact with multiple lag scales simultaneously.

4.1 Full Code + Explanation

MOMENTUM_FEATURES = [
    "lag_1", "lag_2", "lag_3", "lag_5", "lag_10",
    "rolling_mean_5", "rolling_mean_21", "rolling_std_5",
    "rsi_14", "macd_signal",
]

class MomentumAgent(BaseAgent):
    def __init__(self):
        self._model = XGBRegressor(
            n_estimators=200,      # number of boosting rounds (trees)
            learning_rate=0.05,   # shrinkage per tree — smaller = more trees needed, less overfitting
            max_depth=4,          # max depth of each tree — controls complexity
            subsample=0.8,        # 80% of rows sampled per tree — reduces variance
            colsample_bytree=0.8, # 80% of features sampled per tree — like Random Forest feature bagging
            random_state=42,      # reproducibility
            verbosity=0,          # suppress training output
        )

    def _select(self, df: pd.DataFrame) -> pd.DataFrame:
        # List comprehension: keep only columns that exist in this df
        # Defensive: walk-forward folds might have slightly different columns
        return df[[c for c in MOMENTUM_FEATURES if c in df.columns]]

    def fit(self, train_df: pd.DataFrame) -> None:
        self._model.fit(self._select(train_df), train_df["log_return"].values)

    def predict(self, test_df: pd.DataFrame) -> np.ndarray:
        return self._model.predict(self._select(test_df))

4.2 What Is Gradient Boosting? How Does XGBoost Differ from Random Forest?

Random Forest

Trains N decision trees independently and in parallel on bootstrap samples. Each tree votes. Error reduction comes from averaging away variance (bagging). All trees see the same label distribution.

Gradient Boosting (XGBoost)

Trains trees sequentially. Each new tree fits the residual errors (gradient of the loss) of the previous ensemble. Prediction = sum of all trees' outputs × learning rate. Error reduction comes from correcting bias (boosting).

The XGBoost update rule at iteration m:

F_m(x) = F_{m-1}(x) + η · h_m(x)

where h_m(x) fits the negative gradient −∂L/∂F_{m-1}
and η = learning_rate (shrinkage factor)

Why XGBoost dominates tabular ML: It adds L1/L2 regularization on tree weights (unlike vanilla gradient boosting), uses second-order Taylor approximation of the loss for faster convergence, and handles missing values natively.

5. Agent 3 — VolatilityAgent

What it captures: Regime-dependent patterns — the idea that market returns behave differently in high-volatility environments (crashes, rate shocks) vs. low-volatility environments (calm bull markets). The VIX is the key input: it signals the options market's fear level before prices fully react.

5.1 What "Regime-Aware" Means

Financial markets have distinct regimes: calm trending markets, volatile crash environments, and sideways choppy markets. A single model trained across all regimes learns a weighted average of all their behaviors — which performs mediocrely in every regime. VolatilityAgent is designed to detect and respond to regime transitions.

bb_width — realized vol signal

Bollinger Band width measures realized price volatility over 20 days. Narrow = quiet market. Wide = turbulent market. When bb_width suddenly expands, a volatility regime has changed.

rolling_std_5 / 21 — short & medium vol

Two horizons of realized log-return volatility. Divergence between rolling_std_5 and rolling_std_21 signals that a short-term volatility spike is occurring against a calmer backdrop.

vix_level — forward-looking fear

VIX is the options market's implied volatility — forward-looking, not backward-looking. High VIX while price vol is still low = early warning. This is information the other agents cannot see.

vix_change_5d — fear momentum

Is fear rising or falling? VIX spiking rapidly (high vix_change_5d) while already elevated = escalating crisis. VIX falling from high levels = fear subsiding = mean-reversion opportunity.

6. Agent 4 — SequenceAgent (LSTM)

What it captures: Non-linear temporal dependencies that are too complex for tabular models to discover. The LSTM sees raw return sequences and learns patterns like "after 3 days of small losses followed by a sharp drop, the next day tends to recover" — patterns that lag features alone would miss.

6.1 What Is an LSTM?

An LSTM (Long Short-Term Memory) is a type of Recurrent Neural Network (RNN) that maintains a cell state — an internal memory that persists across time steps. Unlike a feedforward network (which processes each input independently), an LSTM processes a sequence and its output at step t depends on all previous inputs.

The critical innovation over vanilla RNNs: LSTMs use gates — learned sigmoid functions that control what information to retain, update, or output at each step. This solves the vanishing gradient problem that made vanilla RNNs fail on long sequences.

6.2 The Three LSTM Gates

🗑 Forget Gate
f_t = σ(W_f·[h_{t-1}, x_t] + b_f)
Decides what to erase from cell state. Output near 0 = forget, near 1 = keep. "Should I forget my understanding of the previous trend?"
➕ Input Gate
i_t = σ(W_i·[h_{t-1}, x_t] + b_i)
c̃_t = tanh(W_c·[h_{t-1}, x_t] + b_c)
Decides what new information to add to cell state. i_t controls how much to add; c̃_t is the new candidate value. "What new pattern should I remember?"
▶ Output Gate
o_t = σ(W_o·[h_{t-1}, x_t] + b_o)
h_t = o_t · tanh(c_t)
Decides what to expose from cell state as the hidden state output. "Based on what I've learned, what prediction do I output?"

The cell state update: c_t = f_t ⊙ c_{t-1} + i_t ⊙ c̃_t — old state (partially forgotten) plus new candidate (partially added). The ⊙ symbol means element-wise multiplication.

6.3 The _LSTMNet Architecture

Input to the network
Input: (batch_size, 30, 1) ← 30 days of returns, 1 value per day
↓
LSTM Layer 1 (hidden=64 units) + dropout(0.2) applied between layers
↓
LSTM Layer 2 (hidden=64 units) ← no dropout after final LSTM layer
↓
Take ONLY the last timestep's output
out[:, -1, :] (batch_size, 64) ← discard steps 0–28, keep step 29
↓
Linear(64 → 1) (batch_size,)
↓
Output: predicted log return for the day AFTER the 30-day window
💡 Why out[:, -1, :]? The LSTM produces one output vector per time step — shape (batch, 30, 64). We only want the prediction for the next day, which is encoded in the final time step's hidden state (-1 = last index). The final hidden state has "seen" all 30 previous days and summarized them into 64 numbers.

6.4 Full Code + Explanation

class _LSTMNet(nn.Module):
    # Prefixed with _ to signal it's a private implementation detail
    def __init__(self, input_size=1, hidden=64, layers=2, dropout=0.2):
        super().__init__()
        self.lstm = nn.LSTM(
            input_size=input_size,    # 1 feature per timestep (the return value)
            hidden_size=hidden,       # 64-dimensional hidden state
            num_layers=layers,        # 2 stacked LSTM layers
            batch_first=True,        # input shape: (batch, seq_len, features) — not (seq_len, batch, features)
            dropout=dropout if layers > 1 else 0.0,  # only apply dropout BETWEEN layers
        )
        self.head = nn.Linear(hidden, 1)   # maps 64-dim hidden state → scalar prediction

    def forward(self, x: torch.Tensor) -> torch.Tensor:
        out, _ = self.lstm(x)          # out shape: (batch, seq_len, hidden=64)
        return self.head(out[:, -1, :]).squeeze(-1)  # take last step → (batch,)


class SequenceAgent(BaseAgent):

    @staticmethod
    def _make_sequences(series: np.ndarray, window: int):
        """Converts a 1-D series into (X, y) sliding window pairs."""
        X, y = [], []
        for i in range(window, len(series)):
            X.append(series[i - window: i])  # window days of history
            y.append(series[i])               # the next day's value (target)
        return np.array(X, dtype=np.float32), np.array(y, dtype=np.float32)

    def fit(self, train_df: pd.DataFrame) -> None:
        returns = train_df["log_return"].values.reshape(-1, 1)

        # StandardScaler: zero mean, unit variance
        # FIT ONLY on training data — never on the full dataset
        scaled = self._scaler.fit_transform(returns).flatten()

        X, y = self._make_sequences(scaled, self.window_size)
        # X shape: (N, window_size) → unsqueeze(-1) → (N, window_size, 1)
        X_t = torch.tensor(X).unsqueeze(-1).to(self._device)
        y_t = torch.tensor(y).to(self._device)

        # 10% validation split for early stopping
        val_n = max(1, int(0.1 * len(X_t)))
        X_tr, X_val = X_t[:-val_n], X_t[-val_n:]
        y_tr, y_val = y_t[:-val_n], y_t[-val_n:]

        loader = DataLoader(TensorDataset(X_tr, y_tr),
                            batch_size=self.batch_size, shuffle=False)
        # shuffle=False: financial time series must stay in temporal order

        self._net = _LSTMNet(input_size=1, hidden=self.hidden).to(self._device)
        optimizer = torch.optim.Adam(self._net.parameters(), lr=self.lr)
        criterion = nn.MSELoss()

        best_val, wait, best_state = float("inf"), 0, None

        for epoch in range(self.epochs):
            self._net.train()
            for xb, yb in loader:
                optimizer.zero_grad()           # clear previous gradients
                loss = criterion(self._net(xb), yb)
                loss.backward()                 # backpropagation through time (BPTT)
                optimizer.step()                # update weights

            # Early stopping check
            self._net.eval()
            with torch.no_grad():
                val_loss = criterion(self._net(X_val), y_val).item()

            if val_loss < best_val - 1e-6:
                best_val, wait = val_loss, 0
                # Save a deep copy of the best weights
                best_state = {k: v.clone() for k, v in self._net.state_dict().items()}
            else:
                wait += 1
                if wait >= self.patience:  # patience=5: stop if no improvement for 5 epochs
                    break

        if best_state:
            self._net.load_state_dict(best_state)  # restore best weights, not final weights

    def predict(self, test_df: pd.DataFrame) -> np.ndarray:
        returns = test_df["log_return"].values.reshape(-1, 1)
        # .transform() (not .fit_transform()) — uses scaler fitted on training data
        scaled = self._scaler.transform(returns).flatten()

        X, _ = self._make_sequences(scaled, self.window_size)
        if len(X) == 0:
            return np.zeros(len(test_df))

        X_t = torch.tensor(X).unsqueeze(-1).to(self._device)
        self._net.eval()
        with torch.no_grad():
            preds_scaled = self._net(X_t).cpu().numpy()

        # De-normalize: inverse_transform converts scaled predictions back to log-return scale
        preds = self._scaler.inverse_transform(preds_scaled.reshape(-1, 1)).flatten()

        # SequenceAgent can only predict from row window_size onward
        # Rows 0..window_size-1 have no full window of history → padded with 0
        full_preds = np.zeros(len(test_df))
        full_preds[self.window_size:] = preds
        return full_preds

6.6 Key Design Details

Why StandardScaler?

LSTM weights are updated via gradient descent. If inputs have very different scales (log returns are ~0.01 while raw prices are ~400), gradients will be inconsistent. Normalizing to zero mean, unit variance stabilizes training dramatically. Fitted on train only to prevent future distribution leakage.

Early Stopping — patience=5

Saves the best model weights whenever validation loss improves. If 5 consecutive epochs pass without improvement, training stops and best weights are restored. Prevents overfitting to training data when the model has already converged.

shuffle=False in DataLoader

Financial time series must remain in temporal order — shuffling would create future leakage within the training loop itself. If batch i contains data from 2018 and batch i+1 from 2015, the LSTM's hidden state carries 2018 information when processing 2015 data.

Why dropout only between layers?

dropout=dropout if layers > 1 else 0.0 — PyTorch's LSTM only applies dropout between LSTM layers, not after the final layer. If you have only 1 layer, there is no "between" — setting dropout on a single-layer LSTM raises a UserWarning and has no effect.

6.7 Vanishing Gradient Problem and How LSTM Solves It

In a vanilla RNN, the gradient of the loss with respect to weights at time step t passes through a chain of multiplications: ∂L/∂W ∝ ∏(∂h_t/∂h_{t-1}). If these Jacobians have singular values < 1, the gradient vanishes exponentially as you go back in time — the network cannot learn dependencies beyond ~10 steps.

The LSTM cell state c_t has a near-direct gradient path: ∂c_t/∂c_{t-1} ≈ f_t. The forget gate f_t ∈ (0,1) controls how much gradient flows back. When the network learns to keep f_t ≈ 1 for important long-range dependencies, gradient can flow back across hundreds of time steps without vanishing.

7. Why 4 Different Agents?

No single model dominates all market regimes. This is the core motivation for the entire multi-agent architecture.

Market ConditionBest AgentWhy
Long bull market (2013–2019)TrendAgentSteady upward drift captured by linear trend component
Momentum-driven rallyMomentumAgentLag features capture serial positive autocorrelation
Volatility spike / crash (COVID 2020)VolatilityAgentVIX spike and Bollinger expansion signal the regime shift
Complex non-linear patternsSequenceAgentLSTM can learn arbitrary temporal dependencies in return sequences
💡 This diversity is what makes the Hedge aggregator powerful If all agents were XGBoost on the same features, they would all fail simultaneously in the same regimes. The Hedge algorithm's regret guarantee only produces strong results when agents have different failure modes. An agent that is wrong when others are right has positive value to the ensemble even if it is wrong more often than not.

Think of it like a portfolio: you want assets that are uncorrelated (or negatively correlated) with each other. A portfolio of 4 identical assets provides zero diversification. A portfolio of 4 different asset classes survives regime changes. Same principle here.

8. Full Interview Q&A

Q1. What is an Abstract Base Class (ABC) and why did you use one here?
An Abstract Base Class is a class that cannot be instantiated directly and defines one or more abstract methods that all subclasses must implement. In Python, it is created by inheriting from ABC and decorating methods with @abstractmethod. I used it to enforce a uniform fit()/predict() interface across all four agents. The Hedge aggregator iterates over a list of agents and calls these methods — it works because every agent is guaranteed by the ABC to have them. The enforcement happens at instantiation time: if you forget to implement predict() in a new agent subclass, Python raises TypeError before you run a single line of code.
Q2. What is gradient boosting, and how does XGBoost differ from Random Forest?
Random Forest trains N trees independently on bootstrap samples and averages their predictions — it reduces variance through bagging. Gradient boosting trains trees sequentially: each new tree fits the negative gradient (residuals) of the current ensemble's loss. XGBoost extends vanilla gradient boosting with second-order Taylor expansion of the loss function (for faster, more accurate splits), L1/L2 regularization on leaf weights (preventing overfitting), column and row subsampling (decorrelating trees), and efficient handling of sparse/missing data. On tabular datasets, XGBoost consistently outperforms Random Forest because it reduces both bias and variance through the sequential correction mechanism.
Q3. Explain the XGBoost hyperparameters you used: n_estimators, learning_rate, max_depth, subsample, colsample_bytree.
  • n_estimators=200: Number of boosting rounds. More trees = better fit on training data but slower and higher overfit risk. 200 is a reasonable default for ~3000 training samples.
  • learning_rate=0.05: Shrinks each tree's contribution by 95%. Small learning rate requires more trees to converge but produces more robust models — no single tree can dominate the ensemble.
  • max_depth=4: Maximum depth of each decision tree. Depth 4 allows up to 4 feature interactions (e.g., "if lag_1 > 0 and RSI < 30 and VIX > 20 and rolling_mean_5 < 0"). Deeper trees risk memorizing noise.
  • subsample=0.8: Each tree is trained on a random 80% of rows. This introduces stochasticity that reduces variance — similar to bootstrap sampling in Random Forest.
  • colsample_bytree=0.8: Each tree uses a random 80% of features. This decorrelates trees — they cannot all rely on the same dominant feature (e.g., lag_1 might dominate if given full access every round).
Q4. What is an LSTM and how does it differ from a simple RNN?
A simple RNN computes h_t = tanh(W·h_{t-1} + U·x_t) — it has one hidden state that is overwritten at every step. The problem: gradients vanish or explode when backpropagating through many time steps, so vanilla RNNs effectively cannot learn dependencies beyond ~10-15 steps. An LSTM maintains two states: a hidden state h_t (short-term working memory) and a cell state c_t (long-term memory). The cell state is protected by learned gates — forget, input, and output — that control what information to erase, write, and read. The cell state update is additive (c_t = f_t ⊙ c_{t-1} + i_t ⊙ c̃_t), which creates a near-direct gradient highway through time, solving the vanishing gradient problem.
Q5. Explain the three gates of an LSTM and what each one controls.
  • Forget gate: f_t = σ(W_f·[h_{t-1}, x_t] + b_f). Output ∈ (0,1) per cell. Value near 0 = erase the corresponding cell memory. Value near 1 = keep it. "Should I reset my understanding of the current market trend?"
  • Input gate: i_t = σ(W_i·[h_{t-1}, x_t]) controls how much of the candidate c̃_t = tanh(W_c·[h_{t-1}, x_t]) to write into the cell state. "Is today's return important enough to update my long-term memory?"
  • Output gate: o_t = σ(W_o·[h_{t-1}, x_t]) controls what portion of the cell state to expose as the hidden state output. h_t = o_t ⊙ tanh(c_t). "Of everything I know, what is relevant for predicting the next return?"
Q6. Why do you normalize returns before feeding them to the LSTM but not to XGBoost?
XGBoost is based on decision trees — trees split on thresholds, and thresholds are invariant to monotonic transformations of the input. Whether lag_1 is 0.005 or 5.0 (scaled by 1000), the tree finds the same optimal split point. Normalization has zero effect on tree-based models. Neural networks, by contrast, use gradient descent to update weights. Gradients are computed as derivatives of the loss with respect to weights, and these depend on the absolute scale of inputs. If log returns (~0.01) enter the LSTM unnormalized, the gradients are tiny and training is slow. Normalizing to unit variance makes gradients well-conditioned and training converges in far fewer epochs. The scaler is fitted on training data only — fitting on the full dataset would leak future distribution information (another form of lookahead bias).
Q7. What is the vanishing gradient problem and how does the LSTM cell state solve it?
In backpropagation through time, the gradient of the loss at time T with respect to parameters at time t involves a product of Jacobians: ∏(∂h_k/∂h_{k-1}) for k from t+1 to T. In a vanilla RNN, each Jacobian involves a matrix with spectral radius (largest singular value) that is typically less than 1. After multiplying many such matrices, the gradient norm decays exponentially — reaching effectively zero for t that are even 10-15 steps back. The LSTM solves this via the additive cell state update: ∂c_T/∂c_t ≈ ∏ f_k (product of forget gate values). When the network learns to set forget gates near 1 for important long-range dependencies, this product stays near 1, and gradients flow freely through time. The key insight is replacing the multiplicative hidden state recurrence with an additive cell state recurrence — the same principle behind ResNets' skip connections.
Q8. What is early stopping, and why use patience=5 specifically?
Early stopping monitors a validation metric (here, MSE on a held-out 10% slice of training data) after each training epoch. When the validation loss fails to improve by more than a threshold (1e-6) for patience=5 consecutive epochs, training stops and the model weights from the best epoch are restored. This prevents overfitting: the training loss continues to decrease while the validation loss plateaus or worsens, a classic sign that the model is memorizing the training set. Patience=5 means we tolerate 5 epochs of no improvement — this gives the optimizer a chance to escape local plateaus before deciding to stop. Very small patience (1-2) stops too aggressively; very large patience (20+) allows too much overfitting before stopping.
Q9. Why does SequenceAgent pad the first window_size=30 predictions with zeros?
The LSTM requires a full window of 30 days to make a prediction. For the first 30 rows of any test period, there are fewer than 30 previous returns available within the test window — so _make_sequences() produces no rows for those positions. Rather than returning a shorter array that would misalign with the other agents' outputs, we pad with zeros. In the Hedge aggregator, zero predictions mean "no signal" — equivalent to predicting the mean return. During walk-forward backtesting, the first fold always has at least 3 years of training data, so the test period is a fresh 3-month window where the LSTM has 30 days of context from the end of training data available. The padding only matters at the very start of the first fold.
Q10. What is DeterministicProcess in statsmodels and why is it useful for financial time series?
DeterministicProcess constructs a design matrix of deterministic (non-random) basis functions of the time index — things like a constant, a linear time trend, quadratic trend, and Fourier sine/cosine terms for seasonality. It is useful for financial time series because markets have structural components that are purely a function of time: the long-run upward drift of equity markets (captured by the linear trend) and calendar-based seasonality effects like the January effect, summer slowdown, and year-end rally (captured by annual Fourier terms). Unlike pure ML models that try to discover these patterns from the data, DeterministicProcess encodes them explicitly, producing interpretable coefficients. The key advantage over hand-crafting these features is that DeterministicProcess correctly handles out_of_sample() — generating the same basis functions for unseen future dates without requiring any future price data.
Q11. Why does TrendAgent use only the date index and not the price or return features?
TrendAgent is designed to capture the structural baseline of markets — the component that would exist even in the absence of short-term noise. If you add price features (RSI, MACD, lags) to TrendAgent, two things happen: first, the model becomes confused about whether to predict trend or momentum — it loses its identity as a specialist. Second, the Fourier/trend signal gets swamped by the noise from price features, producing a model that is worse at capturing trend than a pure trend model. The entire point of the multi-agent architecture is agent specialization: each agent should be the best possible predictor of one specific type of signal. The Hedge aggregator's job is to combine them. A generalist TrendAgent would hurt the ensemble by duplicating the information already in MomentumAgent, reducing the diversity that makes Hedge effective.
Q12. What is colsample_bytree and why does it reduce overfitting?
colsample_bytree controls the fraction of features randomly sampled when building each tree. With colsample_bytree=0.8, each of the 200 trees in the ensemble is built using a random 80% of the available features. This decorrelates the trees: if lag_1 is a dominant predictor, some trees cannot use it at all, forcing them to discover signal in other features. When you average (or in XGBoost's case, sum) many decorrelated trees, variance cancels out — the ensemble is more robust than any individual tree. It is the same mechanism as Random Forest's feature subsampling, applied at the per-tree level. Without it, all 200 trees would tend to split on the same dominant features, producing a highly correlated ensemble that offers no diversity benefit.

w_i(t+1) = w_i(t) · exp(−η · loss_i(t))
then normalize: w_i ← w_i / Σ w_j

Regret bound: R_T ≤ √(T · log(N) / 2) for optimal η = √(log(N) / T)

  • Agents that predict accurately get relatively higher weights
