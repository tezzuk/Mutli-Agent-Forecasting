5. The Lookahead Bias Demo — Most Important Lesson

The lookahead bias demonstration in Part 2 of the notebook is designed to create a visceral, memorable illustration of the problem. Students run two models with identical architectures on identical train/test splits, differing only in one line:

# Correct version
lag1_correct = log_returns.shift(1)    # t-1 predicts t

# Wrong version (lookahead)
lag1_wrong = log_returns                  # t predicts t — the answer is the feature!

What the Results Show

VersionMAEDirectional AccuracyExplanation
Correct (no lookahead)~0.0075~51–53%Genuine signal in lag-1; just barely above 50% coin flip
Wrong (lookahead)~0.0000~100%The feature IS the target — perfect tautological prediction

The contrast is dramatic. A student who sees 100% directional accuracy for the first time will remember this lesson for the rest of their career. The fix is a single method call: .shift(1).

📋 Why This Works As a Pedagogy Technique Showing the wrong result before the correct result is more effective than only showing the correct approach. The cognitive hook — "wait, how is this model achieving 100%?" — creates an active search for the cause. When the student finds it (same-day feature predicts same-day target), the understanding is much deeper than if you'd just told them "use .shift(1)."

6. Why These Specific Exercises?

Exercise: NaiveAgent (always predicts 0)

Purpose: Establishes the correct floor. On daily log returns, sign(0) only matches sign(actual) when actual is exactly 0 — which almost never happens. So NaiveAgent gets ~50% directional accuracy (coin flip). This shows students that 50% is the minimum threshold — anything below means your model is actively hurting you.

What students learn: 50% is not a good score. It's the floor. Their models must beat 50% to be worth anything.

Exercise: YesterdayAgent (predicts lag_1)

Purpose: Tests whether raw momentum is a real signal. If markets have no serial dependence, YesterdayAgent should get exactly 50%. If there's short-term momentum (yesterday up → today up), it gets >50%. If there's mean-reversion (yesterday up → today down), it gets <50%.

What students learn: Empirical investigation of the Efficient Market Hypothesis. (Answer: S&P 500 has slight mean-reversion on daily timescales — YesterdayAgent typically gets ~48–49%.)

Exercise: RSI Contrarian Rule

Purpose: Compares rule-based vs ML. The RSI contrarian rule is simple: if RSI > 70, predict negative; if RSI < 30, predict positive; else predict 0. No training data needed.

class RSIContrarianAgent(BaseAgent):
    def fit(self, train_df): pass   # no training needed

    def predict(self, test_df):
        rsi = test_df['rsi_14'].values
        preds = np.zeros(len(rsi))
        preds[rsi > 70] = -0.001    # overbought → predict down
        preds[rsi < 30] = +0.001    # oversold → predict up
        return preds

What students learn: Rule-based agents are simpler, interpretable, and sometimes competitive with ML. The Hedge algorithm doesn't care whether an agent is ML or rule-based — it only cares about recent prediction quality. Including rule-based agents in the pool is legitimate.

10. Full Interview Q&A — Complete Project

Q1. Walk me through the full architecture of your multi-agent forecasting system.
We start with daily OHLCV data for SPY and VIX from Yahoo Finance via yfinance, covering 2010–2024. A feature engineering layer computes 14 features per day — including log returns (which we use as the prediction target because they're stationary), RSI-14, MACD signal, Bollinger Band Width, 5-day and 21-day rolling statistics, lag features lag_1 through lag_10, and VIX level plus 5-day VIX change. All features use .shift(1) on closing prices to prevent lookahead bias. These features feed four specialist agents: a TrendAgent using linear regression with Fourier seasonality terms, a MomentumAgent using XGBoost on lag and technical features, a VolatilityAgent using XGBoost on VIX and Bollinger features, and a SequenceAgent using a 2-layer LSTM. The Hedge aggregator (Freund & Schapire, 1997) maintains a weight vector over the four agents and updates it after each day's prediction using multiplicative weights. A walk-forward backtest engine evaluates the system on ~44 three-month test windows over the 14-year history. Finally, a Streamlit dashboard visualizes the results with three interactive tabs: forecast vs actual, agent weight evolution, and performance breakdown.
Q2. Why is walk-forward validation the right approach for financial data?
Financial data has a strict temporal structure: the future cannot cause the past. If you randomly split data into train and test sets (like you might with image classification), you allow the model to train on data from 2022 and test on data from 2019 — the model has effectively seen the future during training. This is called temporal leakage. Walk-forward validation respects the time ordering: each test window is strictly in the future relative to its corresponding training window. We use an expanding window — training grows by 3 months each fold, so later folds benefit from more historical data. This matches how a real trading system would be retrained periodically as new data arrives. The result is an unbiased estimate of how the system would have performed if deployed in real time.
Q3. What is the theoretical guarantee of the Hedge algorithm?
The Hedge algorithm satisfies a no-regret guarantee: the total loss of the ensemble over T rounds cannot exceed the total loss of the best individual agent by more than O(√(T·log N)), where N is the number of agents. More precisely, the regret R_T = sum of ensemble losses - min agent's total loss ≤ sqrt(T·log(N)/2). Per round, the excess regret is at most sqrt(log(N)/(2T)), which goes to zero as T grows. This means Hedge is guaranteed to perform at least as well as the best agent in hindsight, plus a vanishing term. The key word is "hindsight" — you don't need to know in advance which agent will be best. Hedge discovers this automatically from the prediction errors.
Q4. What is the difference between RSI and MACD as features?
RSI and MACD capture fundamentally different aspects of price behavior. RSI (Relative Strength Index) measures the ratio of recent gains to recent losses using EWM smoothing. It oscillates between 0 and 100 and is most useful as a mean-reversion signal: RSI above 70 indicates overbought conditions (too many recent up days), RSI below 30 indicates oversold. It's a bounded, normalized measure of short-term directional imbalance. MACD (Moving Average Convergence Divergence) is a momentum signal that measures the gap between a fast EMA (12-day) and a slow EMA (26-day). When the fast EMA is above the slow EMA, short-term momentum is stronger than the long-run trend — a bullish signal. We use the signal line (9-day EMA of MACD), not raw MACD, because the signal line is smoother and less prone to one-day noise spikes. The two features are complementary: RSI catches overbought/oversold extremes; MACD captures the momentum trend direction.
Q5. How does your LSTM agent avoid overfitting?
Three mechanisms: First, dropout — each LSTM layer has a 0.2 dropout rate, which randomly zeroes 20% of neuron activations during training. This prevents any single neuron from becoming essential, forcing the network to learn redundant representations. Second, early stopping — we monitor validation loss with a patience of 5 epochs. If the validation loss doesn't improve for 5 consecutive epochs, training stops and we restore the best model state using .state_dict() and torch.load_state_dict(). This prevents the network from training into a regime where it memorizes training patterns but loses generalization. Third, the walk-forward protocol itself — the LSTM is retrained on each fold's expanding training window. Because training always uses data strictly before the test period, the model cannot memorize future patterns.
Q6. What is your Sharpe ratio and how is it computed?
The Sharpe ratio for our strategy is computed as: mean(strategy_returns) / std(strategy_returns) × sqrt(252), where strategy_returns[t] = sign(predicted_return_t) × actual_return_t. The sign() converts the prediction into a trading signal: +1 for long, -1 for short. We multiply by the actual return to get the strategy's P&L on that day. Annualizing by sqrt(252) converts daily Sharpe to annual Sharpe (252 trading days per year). The interpretation: Sharpe > 1.0 is considered good for a systematic strategy; > 2.0 is excellent. For context, the S&P 500 buy-and-hold has a Sharpe of roughly 0.5–0.7 over long periods. Our Hedge ensemble targets a Sharpe around 0.8–1.2, though exact results depend on the run.
Q7. What would you add if you had more time?
Three high-value additions: First, live mode — fetch today's SPY/VIX from yfinance in the dashboard, use the trained agents to generate tomorrow's prediction, and show the current Hedge weights. This converts the backtest viewer into an actual forecasting tool. Second, transaction cost modeling — real strategies pay bid-ask spread and commissions. Including even a small transaction cost (0.05% per trade) would give a more realistic Sharpe estimate. Third, regime-adaptive η — instead of a fixed η throughout the backtest, adapt η based on market volatility: low volatility → lower η (less urgent to adapt), high volatility → higher η (need to adapt fast). This is a known extension of the Hedge algorithm. Additional agents would also help: a macro agent using economic indicators (PMI, employment data), or a sentiment agent using news embeddings.
Q8. What is the Bollinger Squeeze and why does it matter as a predictor?
The Bollinger Squeeze occurs when Bollinger Band Width reaches a local minimum — the bands are unusually narrow, indicating a period of very low volatility. The empirical observation (Bollinger's original insight) is that periods of low volatility tend to precede large moves. The market "squeezes" before it "pops." As a feature, low Bollinger width gives the VolatilityAgent a heads-up that a regime change may be coming, even though it can't predict the direction. The historical intuition: when uncertainty is low, traders are complacent and not hedging. When uncertainty suddenly rises — because of an unexpected announcement, geopolitical event, or macro surprise — the market moves sharply as participants scramble to adjust. The Bollinger width is essentially measuring the level of complacency in the market.
Q9. How did you prevent lookahead bias in your feature engineering?
The keystone is a single line: close = df["Close"].shift(1). By shifting the close price series forward by one day, every downstream computation uses yesterday's price data, not today's. So when we compute RSI(close), the RSI value for day t uses prices from day t-1 and earlier — never day t's price. The same applies to MACD, Bollinger, and rolling statistics. The lag features are naturally correct: lag_1[t] = return[t-1] by definition. The target column log_return is not shifted — it's what we're predicting. In the feature matrix, each row contains features computed from data available at the end of trading day t-1, and the target is what happens during trading day t. This is the only correct formulation for a live trading system.
Q10. Why XGBoost for some agents and LSTM for others?
Different agents capture different types of signal, and different model families are suited to different signal types. MomentumAgent and VolatilityAgent use tabular features — lag_1 through lag_10, RSI, MACD, Bollinger width. These are well-defined numeric columns with no particular temporal ordering between them once you've computed them. XGBoost handles tabular data efficiently, is robust to outliers (which are common in financial data — extreme events like COVID crash), and provides feature importance out of the box. The SequenceAgent is different — it uses the raw sequence of past 30 returns, and the signal lies in the pattern of the sequence, not individual values. LSTMs are designed specifically to learn from sequences: they maintain a hidden state that can encode long-range dependencies in the time series. No tabular model would capture "the last 30 days had this specific oscillation pattern" the way an LSTM can.
Q11. Explain the Hedge weight update rule from first principles.
We maintain a weight vector w = [w_1, w_2, w_3, w_4] that sums to 1. Initially, all weights are 1/4 (uniform). After each prediction round, we observe the actual return y and compute each agent's squared error L_i = (y_hat_i - y)^2. We update: w_i ← w_i × exp(-η × L_i), then normalize. The exponential exp(-η × L_i) is a multiplier between 0 and 1 — it reduces the weight of agents that made large errors. The larger L_i (bigger error), the smaller exp(-η × L_i), the more that agent's weight is reduced. The normalization step ensures weights still sum to 1. The learning rate η controls the severity of the penalty: large η → aggressive downweighting after errors; small η → gradual adaptation. The key intuition: we never zero out any agent. exp() is always positive, so every agent retains some weight. A bad agent in a bull market might be exactly right in a bear market — keeping it in the pool allows it to recover.
Q12. What makes this a "multi-agent" system vs just an ensemble?
A standard ensemble (random forest, gradient boosting) combines many similar models — all trained on the same features, all using the same algorithm — to reduce variance. Each model in the ensemble is essentially a perturbed version of the same approach. Our multi-agent system is architecturally different: each agent is a specialist that looks at a fundamentally different aspect of the market. The TrendAgent doesn't look at RSI or lag features. The VolatilityAgent doesn't look at Fourier terms. Each agent has a distinct economic theory behind its feature set — the trend agent embodies "prices have a long-run upward trend," the momentum agent embodies "recent returns predict near-term returns," the volatility agent embodies "stress regimes require different models." The Hedge aggregator then discovers, in real time, which economic theory is currently most predictive. This is closer to a committee of domain experts than to bootstrapped samples of the same model.
Q13. How would you deploy this in production?
For a real deployment: First, replace yfinance with a paid data provider (Bloomberg API, Refinitiv, Polygon.io) — yfinance has reliability issues and delayed data. Second, move the backtest and agent training to a scheduled job (cron, Airflow, or a cloud scheduler) that runs nightly after market close. Third, serialize the trained agents and current Hedge weights to disk after each training run using joblib for sklearn/XGBoost models and torch.save() for LSTM. Fourth, have a separate inference service that loads the serialized models and computes tomorrow's prediction at market close each day. Fifth, add monitoring: log every prediction, actual result, and weight update to a database so you can detect if any agent's performance degrades. Sixth, add risk limits: if the predicted return magnitude is outside 3 standard deviations of historical predictions, reject the signal. Finally, paper trade for 3–6 months before committing real capital — even a working backtest can have structural differences from live trading (market impact, execution delays, data quality).
Q14. What are the limitations of your approach?
Several honest limitations: First, transaction costs are not modeled. Our strategy trades long/short daily, which implies high turnover. Real spreads and commissions would erode returns substantially. Second, no position sizing — we use unit signals (+1 or -1). A real strategy would size positions based on prediction confidence and portfolio risk. Third, the LSTM's performance on financial data is debatable — LSTMs on daily returns are prone to overfitting and have shown mixed results in academic literature. The tabular agents may be more reliable. Fourth, survivorship bias — using SPY avoids the issue, but if we extended to individual stocks, we'd need to ensure our universe includes delisted companies. Fifth, the Efficient Market Hypothesis suggests daily return prediction is close to impossible. Our results (51–54% directional accuracy) are at the edge of what can be achieved — and those numbers may not hold going forward if market microstructure changes. The project is an excellent educational demonstration of the pipeline, but should not be mistaken for a profitable trading strategy without much more rigorous validation.
Q15. What did you teach students in the Week 3 notebook and what did they learn?
Week 3 introduced the domain-specific layer between general ML (Weeks 1–2) and the pipeline (Weeks 4–8). Students learned: why financial return prediction requires stationary targets (log returns vs prices), illustrated by plotting both series; the lookahead bias problem, demonstrated empirically by showing a model achieves 100% accuracy when you accidentally feed it the current-day price, dropping to realistic ~52% when fixed; three technical indicators (RSI, MACD, Bollinger) implemented from scratch rather than from a library, so students understand what the numbers mean, not just how to call .ta.RSI(); the Abstract Base Class pattern for enforcing a uniform agent interface; a full MomentumAgent implementation in XGBoost; and the Hedge algorithm's update rule, including a numerical η sensitivity experiment. The exercises (NaiveAgent, YesterdayAgent, RSI contrarian rule) test synthesis rather than replication — students must apply the concepts to problems they haven't seen, which is the only reliable test of understanding.
