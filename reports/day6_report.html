2. Why Streamlit Over Jupyter?

📖 Jupyter = Learning Tool

Notebooks are for step-by-step exploration. To show someone a Jupyter notebook, they need Python installed, the right kernel, and to run cells in order. One wrong import breaks everything. It's a development environment, not a product.

🌐 Streamlit = Demo Product

One command: streamlit run app.py. Opens in the browser. Anyone can interact with it — no Python knowledge needed. A recruiter, professor, or client can use it on their laptop without touching a terminal.

🎯 The Demo Factor

A working interactive app is 10× more impressive in an interview than showing screenshots of a notebook. You can drag the date slider, toggle agents, and watch the weight chart update in real time. That's a conversation, not a presentation.

☁ Free Deployment

Streamlit Community Cloud deploys directly from a GitHub repo — free, no credit card, live URL in minutes. You can share a link with anyone in the world. That's not possible with Jupyter without significant infrastructure.
✅ vs. Dash / React Dash (Plotly) requires more boilerplate (callbacks, layout objects) and is harder to prototype quickly. React requires JavaScript and a frontend build pipeline. Streamlit is pure Python — write it like a script, get an interactive app. For a 7-day project, Streamlit is the correct choice.

3. @st.cache_data — How It Works

The problem: Streamlit re-runs the entire script from top to bottom on every user interaction. Every checkbox click, every date range change triggers a full re-execution. Without caching, load_results() reads 3,500 rows from disk on every interaction. Multiplied across multiple charts — this makes the app feel sluggish.

# WITHOUT caching:
def load_results():
    return pd.read_csv(RESULTS_PATH, ...)   # runs on EVERY interaction → slow

# WITH caching:
@st.cache_data
def load_results():
    return pd.read_csv(RESULTS_PATH, ...)   # runs ONCE, cached in memory after

How the cache works internally

  1. First call: Streamlit computes the result and stores it in an in-memory dict, keyed by the function name + arguments.
  2. Subsequent calls with same arguments: Streamlit returns the cached copy instantly (microseconds vs. milliseconds).
  3. Cache invalidation: occurs when the function's source code changes, or when you call st.cache_data.clear().
  4. Memory: cached DataFrames live in RAM. For large datasets, use @st.cache_resource instead (shared object, not per-session copy).
💡 Why the backtest doesn't re-run on slider moves The backtest results are pre-computed and saved to CSV (Day 5). The dashboard only reads that CSV — it never re-runs the 44-fold walk-forward engine. This is intentional: the backtest takes several minutes; the dashboard should respond in milliseconds. Separation of concerns: compute offline, display online.

4. Sidebar Design

# Date range — filters the entire results DataFrame
date_range = st.sidebar.date_input(
    "Date range",
    value=(results.index.min().date(), results.index.max().date()),
    min_value=results.index.min().date(),
    max_value=results.index.max().date(),
)
if len(date_range) == 2:
    results = results.loc[str(date_range[0]): str(date_range[1])]

# Agent checkboxes — builds the active_agents list
active_agents = [a for a in AGENT_NAMES
                 if st.sidebar.checkbox(a, value=True, key=f"chk_{a}")]

Why len(date_range) == 2 guard? When the user is mid-selection (clicked start date but not end date yet), st.date_input returns a tuple of length 1. Applying the filter with only one date would crash. The guard waits until both dates are selected.

Why not an η slider? η was fixed at 0.1 when the backtest ran. A live η slider would imply the Hedge aggregator is re-running in real time — which it isn't. Showing it as an info panel is honest: it tells the user what was used, without implying they can change it post-hoc.

Design Decisions

  • visible="legendonly": The trace exists in the legend but doesn't render in the chart until the user clicks it. This avoids a cluttered chart with 6 overlapping lines while keeping all agents accessible. The user self-selects what they want to see.
  • opacity=0.5 for individual agents: When shown, they appear visually subordinate to the ensemble, which is the primary signal of interest.
  • Color coding is consistent across all tabs: TrendAgent is always blue (#4361ee), MomentumAgent is always orange (#f77f00), etc. This allows instant cross-tab visual matching without re-reading legends.

Regime Bands

def add_regime_bands(fig, df):
    for r in REGIME_BANDS:
        # Only add band if it falls within the current date range
        if r["x0"] >= str(df.index[0]) and r["x1"] <= str(df.index[-1]):
            fig.add_vrect(x0=r["x0"], x1=r["x1"],
                          fillcolor=r["color"], line_width=0,
                          annotation_text=r["label"],
                          annotation_position="top left")
    return fig

add_vrect() draws a vertical shaded rectangle spanning the full y-axis. line_width=0 removes the border — only the fill color shows. The date range guard prevents rendering a band that's outside the filtered window (which would cause a Plotly layout error).

6. Tab 2 — Agent Weights: The Key Visual

This is the most important chart in the entire dashboard. It shows the Hedge algorithm doing its job in real time — redistributing trust among agents as the market changes.

fig2 = go.Figure()
for i, agent in enumerate(wdf.columns):
    fig2.add_trace(go.Scatter(
        x=wdf.index, y=wdf[agent],
        name=agent,
        # First trace fills to zero; subsequent traces fill to previous trace
        fill="tozeroy" if agent == wdf.columns[0] else "tonexty",
        line=dict(color=AGENT_COLORS.get(agent, "#999"), width=0.5),
        stackgroup="one",    # critical: all same group → stacked
    ))
fig2.update_layout(yaxis=dict(tickformat=".0%", range=[0, 1]))

How Stacking Works

stackgroup="one" tells Plotly that all traces in this group should be stacked vertically. Internally, Plotly accumulates the y values: if Agent A has weight 0.4 and Agent B has weight 0.3, Agent B's line is drawn at 0.7 (not 0.3). The gap between lines represents each agent's weight. Because all weights sum to 1 at every point, the total filled area always reaches exactly 1 (100%).

What to look for in the weights chart:

  • COVID Crash (Feb–Apr 2020): VolatilityAgent's band should widen — it uses VIX features that spike in crises, making it temporarily more accurate than the others.
  • Calm periods (2012–2019): MomentumAgent or TrendAgent should dominate — these agents work best in trending, low-volatility markets.
  • Rate Hike Cycle (2022): TrendAgent may gain weight — it's designed to capture persistent directional trends, and 2022 was a persistent downtrend.
  • Rapid fluctuations: If weights jump erratically every few days, η=0.1 may be too large. The optimal η≈0.0094 would produce smoother evolution.

7.3 Equity Curves

equity(t) = $10,000 × exp( Σ sign(pred_i) × actual_i ) for i = 1 to t

Each day: if pred > 0, go long (earn +actual). If pred < 0, go short (earn −actual). This models a simple dollar-neutral long/short strategy with no leverage and no transaction costs.

What to expect: TrendAgent may gain weight — its DeterministicProcess captures persistent directional trends. VolatilityAgent may contribute less than in 2020 (VIX was elevated at ~25–30, not extreme at 80+).

10. How Streamlit's Execution Model Works

Streamlit's programming model is unusual compared to traditional web frameworks. Understanding it is important for building reliable apps.

The Reactive Re-run Model

  1. Every interaction triggers a full script re-run. When you move a slider or click a checkbox, Streamlit re-executes app.py from line 1 to the end.
  2. @st.cache_data skips expensive computations. Functions decorated with @st.cache_data check if they've been called with the same arguments before. If yes, they return the cached result without executing the function body.
  3. st.stop() is an early exit. When called, it halts execution at that point. We use it when results is None — no point rendering 150 more lines of chart code if the data doesn't exist.
  4. st.tabs() is a context manager list. Code inside with tab1: only renders in Tab 1. Streamlit still executes it during every re-run but only displays it when that tab is active.
  5. Session state persists across re-runs. st.session_state (not used here) allows storing values that survive a re-run. For example, if you wanted to accumulate user clicks over time.
📋 Why widget keys matter Every interactive widget needs a unique key parameter. In our agent checkboxes: key=f"chk_{a}". Without unique keys, Streamlit cannot distinguish between widgets of the same type and will raise a DuplicateWidgetID error.

11. Full Interview Q&A

Q1. What is Streamlit and how does it work?
Streamlit is a Python library that converts a regular Python script into an interactive web application. It works through a reactive re-run model: every user interaction (slider, checkbox, button) triggers a full re-execution of the script. Streamlit tracks widget state between runs so the widgets retain their values. The result is that you write procedural Python code — no callbacks, no event handlers — and get an interactive web app with no frontend engineering.
Q2. Why did you choose Streamlit over Dash or a React frontend?
Streamlit is pure Python — no JavaScript, no callback decorators, no layout object hierarchies. For a 7-day research project where the goal is a working demo, not a production SaaS, Streamlit gets you from code to interactive app in the shortest time. Dash requires Plotly callback decorators and a more explicit layout system. React requires a full JavaScript frontend build pipeline. Streamlit's limitation is that it's less flexible for complex UI interactions — if we needed custom JavaScript components or complex state machines, we'd move to Dash or React. But for showing research results, Streamlit is the right tool.
Q3. What does @st.cache_data do and why is it essential here?
@st.cache_data memoizes function results keyed by the function's source code and input arguments. On the first call, it executes the function and stores the return value in memory. On subsequent calls with the same arguments, it returns the cached value without re-running the function. It's essential here because every user interaction re-runs the entire script. Without caching, load_results() reads a 3,500-row CSV from disk on every checkbox click. With caching, that read happens once per session. The cache invalidates automatically when the function's source code changes.
Q4. What is stackgroup in Plotly and how does it create the stacked area chart?
stackgroup is a parameter on go.Scatter that groups traces for stacking. All traces with the same stackgroup string are accumulated: each trace's y values are added on top of the previous trace's values. Combined with fill="tozeroy" (first trace fills down to zero) and fill="tonexty" (subsequent traces fill to the previous trace's line), this creates a stacked area chart where the total filled area at each x equals the sum of all y values. Since agent weights always sum to 1, the total area always reaches exactly 100%. The visual result is that each colored band represents one agent's current weight share.
Q5. Why are individual agent lines visible="legendonly" by default?
With 4 agents + Hedge ensemble + actual returns, that's 6 overlapping time series on one chart. Daily financial return series are noisy — 6 noisy lines overlap into an unreadable mass. Setting individual agents to visible="legendonly" hides them by default while keeping them in the legend. Users can click any agent's legend entry to show it. This progressive disclosure pattern lets the chart focus on the two primary signals (actual and ensemble) while making individual agents accessible on demand.
Q6. What does the stacked area chart in Tab 2 tell you that a line chart can't?
A stacked area chart with weights that sum to 1 shows the composition of the ensemble at every point in time. A line chart would show four weight trajectories that may cross and overlap — hard to read, and it doesn't convey the constraint that weights must sum to 1. The stacked area makes it visually obvious: when VolatilityAgent's band expands, some other agent's band must shrink by exactly the same amount. You can see the Hedge algorithm redistributing trust in real time as a visual composition, not just four separate numbers.
Q7. What are the two regime periods you annotated and why those specifically?
COVID Crash (Feb 20 – Apr 7, 2020): SPY fell 34% in 33 days, VIX hit 82. The fastest bear market in history and the single most extreme volatility event in our 2010–2024 window. This tests whether VolatilityAgent's VIX features provide early warning and whether the ensemble's weight shifts are visible. Rate Hike Cycle (Jan – Dec 2022): The Fed raised rates from 0.25% to 4.25%, SPY fell 19%. Unlike COVID (shock), this was a persistent macro-driven downtrend over a full year — a very different regime type. These two events together stress-test different agent specializations: shock vs. persistent trend.
Q8. Explain the equity curve formula 10_000 * np.exp(np.cumsum(np.sign(preds) * actuals)).
sign(preds) is the trading signal: +1 long or -1 short. Multiplying by actuals gives the daily strategy return: positive when the prediction direction is correct, negative when wrong. cumsum takes the running total of daily log returns — this is the cumulative log return from the start of the backtest to day t. exp() converts cumulative log return to a price ratio: if cumsum=0.5, the portfolio has grown by e^0.5 ≈ 1.65×. Multiplying by 10,000 converts to dollar terms starting from $10,000. The result is a continuous equity curve that compounds daily returns correctly using log-return arithmetic.
Q9. Why is the 50% directional accuracy baseline line important?
Without the baseline, a bar showing 52% directional accuracy looks unimpressive — it's "just above 50%." With the baseline visible, the viewer immediately understands that 50% is what a random coin-flip strategy achieves. On daily equity returns, where professional fund managers struggle to consistently exceed 54%, even 52% represents real predictive skill when sustained over 14 years and 3,500 trading days. The baseline contextualizes the number. Without it, the chart is misleading by omission.
Q10. How would you deploy this dashboard for free?
Streamlit Community Cloud — push the repo to GitHub, connect the repo in share.streamlit.io, and it deploys automatically. The app gets a public URL like yourname-appname.streamlit.app. It's free with no credit card required. The only requirement is that requirements.txt is in the repo root. The limitation is it sleeps after 24 hours of inactivity (re-wakes on first visit) and has resource limits (1GB RAM). For our backtest CSV being pre-generated, this is fine — the dashboard only reads data, never computes the backtest at runtime.
Q11. If a recruiter can't run Python, how would you show them this dashboard?
Three options in increasing order of effort: (1) Deploy to Streamlit Community Cloud and share the URL — they visit it in a browser like any website. (2) Record a short screen capture (Loom or similar) showing the dashboard in action — date range changes, tab switching, weight evolution chart. (3) Take the most visually striking screenshot (the stacked area weight chart with regime bands) and include it in the resume or portfolio. Option 1 is ideal because they can interact with real data. A live app link on a resume is memorable in a way that a GitHub link to code is not.
Q12. What would you add if you had more time?
Three high-value additions: (1) Regime selector — filter the metrics table and equity curve by period (calm / crisis / trending) to show which agents dominate each regime. (2) Live mode — fetch today's SPY/VIX data via yfinance, use the trained agents to make tomorrow's prediction, show the current Hedge weights. This turns the dashboard from a backtest viewer into an actual forecasting tool. (3) η sensitivity panel — let the user choose η and re-run the aggregator (not the agents — just the Hedge weighting) on the pre-computed predictions. This is fast enough to be interactive and shows the η trade-off visually.
