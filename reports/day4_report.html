2. Why a Dedicated Evaluation Harness?

Running code and evaluating code are different things. A dedicated evaluation harness enforces scientific discipline — it ensures every agent is tested under identical conditions, with the same train/test split, the same metrics, and a documented record of results. Without this, it is easy to accidentally cherry-pick good results or miss subtle bugs (like training and evaluating on the same window).

📋 Reproducibility

Every run of evaluate.py produces the same comparison table. The output is saved to results/agent_comparison_2019.csv — a permanent record you can cite in a presentation or paper.

📈 Two Metrics, Not One

MAE measures regression accuracy. Directional Accuracy measures trading utility. Both are needed. A model with low MAE but 49% DA is useless for trading; a model with 53% DA but slightly higher MAE is valuable.

⚠ The 2× Sanity Check

The LSTM is complex and slow to train. A hard gate — "LSTM must not be >2× worse than MomentumAgent" — catches normalization bugs, insufficient epochs, or architectural issues before they propagate into the backtest.

🔍 Feature Importance

XGBoost's built-in gain importance tells you which features actually drove the model's predictions. This is essential for debugging and for explaining the model to an interviewer or client.

3. Metric 1 — MAE (Mean Absolute Error)

MAE = (1/n) · Σ |y_true_i − y_pred_i|

MAE measures the average magnitude of prediction errors in the same units as the target. Here, the target is log returns — so MAE = 0.006 means on average you are off by 0.6 percentage points per day.

Why MAE for Regression?

  • Interpretable units: 0.006 log return = ~0.6% — directly meaningful.
  • Robust to outliers: unlike RMSE (which squares errors and over-penalizes crash days), MAE treats all errors proportionally.
  • Standard baseline: the naive benchmark is MAE of a model that always predicts 0 (the mean of log returns ≈ 0). Any model should beat this.
💡 MAE's Limitation MAE doesn't capture direction. A model that predicts +0.005% when the actual is −0.005% has MAE of 0.01 — worse than a model predicting 0.0 (MAE = 0.005). But the first model correctly called the direction (positive), while predicting 0.0 has no directional signal at all. This is why Directional Accuracy must be reported alongside MAE. 
from sklearn.metrics import mean_absolute_error

mae = mean_absolute_error(y_true, y_pred)
# sklearn computes: np.mean(np.abs(y_true - y_pred))
# Result: a single float in the same units as the target (log returns)

4. Metric 2 — Directional Accuracy

Directional Accuracy = mean( sign(y_true) == sign(y_pred) )

Directional accuracy asks: "Did you correctly predict whether the market went up or down?" For a trading system, this is often more important than the exact magnitude — if you call the direction right, you profit. If you call it wrong, you lose.

Why 50% Is the Random Baseline

For any symmetric distribution of returns (which daily log returns approximately are), the market goes up roughly half the time and down roughly half the time. A coin flip (or always predicting +0.001) would get ~50% directional accuracy purely by chance. Therefore:

  • 50%: random / no signal
  • 52–53%: meaningful edge on daily data — this is what professional quants target
  • >55%: strong signal (rare in practice; markets are efficient)
💡 Why 52% Is Actually Meaningful Over 252 trading days, a 52% DA model makes correct calls on ~131 days and wrong calls on ~121 days — a +10 day edge. If each correct call earns a small positive return, this compounds to a statistically significant Sharpe ratio over many years. The Kelly criterion shows that even a 52% edge justifies a positive bet size.

How np.sign() Works

np.sign(0.003)   # → +1.0  (positive return → predict long)
np.sign(-0.002)  # → -1.0  (negative return → predict short)
np.sign(0.0)     # →  0.0  (exactly zero — rare, treated as neutral)

# Directional accuracy implementation:
def directional_accuracy(y_true, y_pred):
    return float(np.mean(np.sign(y_true) == np.sign(y_pred)))
# Returns a float between 0.0 and 1.0
# 0.5 = random, 0.53 = good, 0.55+ = excellent

6.2 The Sliding Window Mechanism

The _make_sequences(series, window=30) function converts a 1-D return series into overlapping input-output pairs. Here is how it works for a short example with window=3:

Input series: [r0, r1, r2, r3, r4, r5, r6]

Window=3 sliding sequences:
X[0] = [r0, r1, r2] → y[0] = r3
X[1] = [r1, r2, r3] → y[1] = r4
X[2] = [r2, r3, r4] → y[2] = r5
X[3] = [r3, r4, r5] → y[3] = r6

With window=30 and N=3520 rows → 3490 training sequences

Each row of X is a 30-day history of returns. The model must predict the next day's return y from that history. The windows overlap — every day appears in up to 30 different sequences as part of the input.

6.3 Early Stopping — Step by Step

What patience=5 means: "If the validation loss has not improved for 5 consecutive epochs, stop training and restore the best weights seen so far."

Trace through a training run:

  • Epoch 1: val_loss = 0.00012 → new best. Save state. wait=0.
  • Epoch 2: val_loss = 0.00010 → new best. Save state. wait=0.
  • Epoch 3: val_loss = 0.00011 → no improvement. wait=1.
  • Epoch 4: val_loss = 0.00013 → no improvement. wait=2.
  • Epoch 5: val_loss = 0.00012 → no improvement. wait=3.
  • Epoch 6: val_loss = 0.00014 → no improvement. wait=4.
  • Epoch 7: val_loss = 0.00015 → no improvement. wait=5 = patience → STOP.
  • Restore weights from Epoch 2 (best_val = 0.00010).

6.4 Why .clone() For Best State Saving

# WRONG — saves a reference, not a copy
best_state = self._net.state_dict()
# state_dict() returns a dict of tensors that are still LIVE in the model.
# As training continues, the tensors change. best_state now reflects
# the CURRENT weights, not the best weights. You've lost the snapshot.

# CORRECT — saves an independent deep copy
best_state = {k: v.clone() for k, v in self._net.state_dict().items()}
# .clone() creates a new tensor with the same values but independent memory.
# Subsequent training steps don't affect best_state.

6.5 Why LSTM Needs StandardScaler But XGBoost Does Not

LSTM (Gradient-Based)

LSTM learns via backpropagation — gradients flow through weights. If inputs are on very different scales (e.g., returns ranging from -0.12 to +0.09), gradients become unstable. Large inputs → large activations → saturation of sigmoid/tanh gates → vanishing gradients. StandardScaler centers inputs to mean=0, std=1, putting them in the healthy activation range.

XGBoost (Tree-Based)

XGBoost splits on threshold values: "if lag_1 > 0.003, go left." The split threshold is always a value from the data — it doesn't matter if lag_1 is in range [−0.12, 0.09] or [−12, 9]. Trees are scale-invariant by design. StandardScaler would have zero effect on XGBoost's predictions.

6.6 Why shuffle=False in DataLoader

⚠ Never shuffle time series sequences Each sequence in X is [r_{t-30}, ..., r_{t-1}] → predict r_t. These sequences have temporal structure — the patterns the LSTM learns (momentum, mean-reversion) exist across consecutive time steps. If you shuffle, sequence 0 (from early 2010) and sequence 500 (from 2012) can be in the same mini-batch. The LSTM has no way to maintain a coherent hidden state across temporally distant data. More importantly, shuffling with sliding windows causes train/test contamination: if sequence [r_{100}...r_{129}] is in the test set, sequences [r_{98}...r_{127}] and [r_{99}...r_{128}] (which share most of the same data) can leak into training.

full_preds = np.zeros(len(test_df))

7. XGBoost Feature Importance

7.1 What feature_importances_ Measures

XGBoost's default feature_importances_ is based on gain — the total reduction in loss (MSE in our case) across all decision tree splits that used that feature. The higher the gain, the more that feature contributes to reducing prediction error.

Gain vs. Frequency Importance:

  • Frequency: how many times a feature is used as a split point. Can be misleading — a feature split many times on low-information splits scores high.
  • Gain: total loss reduction from all splits on that feature. A feature used once on a highly informative split outscores a feature used 100 times on trivial splits.
  • Gain is more informative for understanding which features actually drive model performance.
def feature_importance(self) -> pd.Series:
    cols = [c for c in MOMENTUM_FEATURES]
    return pd.Series(
        self._model.feature_importances_,  # numpy array, one value per feature
        index=cols[:len(self._model.feature_importances_)]
    ).sort_values(ascending=False)        # highest importance first

9. Why LSTM Doesn't Always Beat XGBoost on Financial Tabular Data

This is a widely misunderstood topic. Deep learning has transformed computer vision, NLP, and audio — but on tabular data with limited samples, gradient-boosted trees frequently outperform neural networks. Understanding why is essential for any ML practitioner.

Reason 1: Sample Size

Our training set has ~2,200 rows (2010–2018). An LSTM with two layers of 64 units has ~66,000 parameters. That is a 30:1 parameter-to-sample ratio — severely underdetermined. XGBoost with n_estimators=200 has far fewer effective parameters per data point and benefits from strong regularization via subsample and colsample_bytree.

Reason 2: Inductive Bias

XGBoost has a strong inductive bias toward axis-aligned decision boundaries with multiplicative feature interactions. This is well-suited to financial data, where features like "RSI > 70 AND lag_1 < 0" are natural predictive patterns. LSTMs assume sequential temporal dependency — which is true for language but weaker for daily equity returns, where the market is close to a random walk.

Reason 3: The Value of LSTM Is Diversity, Not Raw Accuracy

✅ Key Insight for the Hedge Algorithm Even if SequenceAgent has slightly higher MAE than MomentumAgent, it still contributes value to the ensemble — because it captures different patterns. When MomentumAgent is wrong (e.g., during a regime change), SequenceAgent may be right. The Hedge algorithm exploits exactly this complementarity. An ensemble of diverse mediocre models often beats a single excellent model in financial forecasting.

When LSTM Would Shine

  • Higher-frequency data (minute-by-minute returns) with thousands of patterns
  • Multi-asset input (stock returns + news sentiment + order flow simultaneously)
  • Longer sequences (90-day or 252-day windows) on stationary features
  • With 10,000+ training examples (e.g., using multiple assets as training instances)

10. Full Interview Q&A

Q1. What is directional accuracy and why is it more useful than MAE for a trading system?
Directional accuracy measures the fraction of predictions where the predicted sign matches the actual sign — did you correctly call up or down? For a trading system, this is more actionable than MAE because P&L is driven by direction: if you predict up and the market goes up, you profit; if you predict down and it goes up, you lose. A model with excellent MAE that's wrong about direction 52% of the time loses money. A model with slightly worse MAE but 53% DA generates positive returns. That said, both metrics together give a complete picture — DA alone ignores magnitude, which matters for position sizing.
Q2. Why is 50% the random baseline for directional accuracy, not some other number?
For daily equity log returns, the distribution is approximately symmetric around zero — the market goes up roughly as often as it goes down over any given year. A model that always predicts +0.001 (or a coin flip) would correctly call the direction about 50% of the time purely by chance, without learning anything. Therefore 50% is the floor. Anything above 50% represents genuine predictive skill. On very long-term upward trending data (e.g., 20-year periods), a "always predict up" model might achieve 53–54% — but this is not skill, it's just riding the trend. To isolate skill, you compare against the directional accuracy of a "predict the historical average direction" naive model.
Q3. Why do you use shuffle=False in your LSTM DataLoader?
Two reasons. First, the LSTM's hidden state is designed to process sequential patterns — shuffling destroys the temporal structure that makes LSTM different from a simple feedforward network. If epoch 3 trains on sequences from 2015 interspersed with 2018, the model never learns coherent multi-step temporal dynamics. Second, our sliding windows overlap: sequence i = [r_{i}...r_{i+29}] and sequence i+1 = [r_{i+1}...r_{i+30}] share 29 of 30 data points. If shuffled, these near-identical sequences can land in both train and validation splits, causing evaluation contamination. Setting shuffle=False preserves temporal order and prevents this.
Q4. Why does XGBoost not need StandardScaler but LSTM does?
XGBoost is tree-based — it makes decisions by comparing a feature value to a threshold: "if lag_1 > 0.003, go left." The threshold is always derived from the actual data values, so the absolute scale is irrelevant. Multiplying all features by 100 would change the threshold to 0.3 but the split would be identical. LSTM, by contrast, learns via gradient descent. Input values flow through sigmoid and tanh gates: if inputs are large (e.g., log returns scaled to be large numbers), the gate activations saturate near 0 or 1, killing gradients — the vanishing gradient problem. StandardScaler ensures inputs have mean=0 and std=1, keeping activations in the linear range where gradients are healthy.
Q5. What is "gain" feature importance in XGBoost and how does it differ from "frequency"?
Gain importance sums the total reduction in the loss function (MSE in our case) across all decision splits on that feature, across all trees. A feature that appears in one critical split that halves the MSE gets a high gain score. Frequency importance counts how many times a feature appears as a split point regardless of the quality of those splits. Frequency is misleading: a useless feature that generates many low-quality splits scores high on frequency but near zero on gain. Gain is more meaningful for understanding which features actually drive model performance. In practice, for our MomentumAgent, lag_1 typically shows high gain because recent return is the single most predictive feature for next-day return.
Q6. What does your 2× sanity check tell you about the LSTM?
It's a debugging gate. If the LSTM's MAE is more than twice the MomentumAgent's MAE, something is fundamentally broken — not just underperforming due to the inherent difficulty of the prediction task. Possible causes: the StandardScaler was accidentally fitted on the full dataset (including test), introducing lookahead into the normalization; the model didn't train for enough epochs and still has near-random weights; the sequence length of 30 is too long for the amount of training data we have, causing underfitting. A ratio under 2× means the LSTM is competitive — even a ratio of 1.5× is acceptable because the LSTM contributes ensemble diversity even with slightly worse raw accuracy.
Q7. What is early stopping and why does LSTM need it but XGBoost doesn't?
Early stopping halts training when the model stops improving on held-out validation data, then restores the best weights seen. LSTM needs it because neural networks can overfit: with enough training epochs, they memorize the training sequences and their validation performance degrades even as training loss decreases. XGBoost with the hyperparameters we use (subsample=0.8, colsample_bytree=0.8, max_depth=4) is already heavily regularized — the tree structure and subsampling prevent the kind of variance explosion that requires early stopping. Additionally, XGBoost is trained in a single pass through boosting rounds (not iterative gradient descent), so there's no notion of "epochs" during which overfitting progressively worsens.
Q8. Why do you save best_state with .clone() instead of just a reference?
state_dict() in PyTorch returns a dictionary where the values are tensor objects that share memory with the model's live parameters. If you store this dict directly, the "snapshot" updates in place as training continues — you end up with the final weights, not the best weights. .clone() creates a completely independent copy of each tensor in separate memory. Subsequent gradient updates don't affect cloned tensors. At the end of training, we call self._net.load_state_dict(best_state) to restore the actual best checkpoint. This pattern — save-clone on improvement, restore at end — is standard in PyTorch early stopping implementations.
Q9. What would happen if you shuffled time series before creating LSTM sequences?
Catastrophic data leakage. Consider a training set of [r_1, ..., r_3000] and a test set of [r_3001, ..., r_3520]. If you shuffle the full series first, then create sliding windows, sequence [r_2990...r_3019] overlaps with the test set boundary — part of the input is from training, part from test. The model has seen future data. More subtly, even within the training set, shuffling before windowing causes near-identical sequences (which share 29 of 30 points) to land in both train and validation folds. The model appears to validate well because it has already memorized the overlapping sequences. In practice, this inflates validation DA by 5–10 percentage points, leading you to believe you have a better model than you do.
Q10. If SequenceAgent fails the 2× check, what are the first three things you'd debug?
  1. Check StandardScaler fit scope. Print scaler.mean_ and scaler.scale_ before and after fitting. If the scaler was accidentally fit on test data too (e.g., by calling fit_transform on the full DataFrame), the predictions will be incorrectly de-normalized. Fix: ensure scaler.fit_transform(train_returns) then scaler.transform(test_returns).
  2. Check training convergence. Add print(f"Epoch {epoch}: val_loss={val_loss:.6f}") to the training loop. If val_loss is still decreasing at epoch 20, increase epochs to 50. If val_loss immediately stops improving (from epoch 2–3), the model is stuck — try a lower learning rate (1e-4 instead of 1e-3).
  3. Reduce window size. Try window_size=15 or window_size=20. With only ~2,200 training rows, window=30 gives ~2,170 sequences — borderline for an LSTM. A shorter window means more sequences and simpler patterns to learn.
Q11. Why is 52% directional accuracy meaningful even though it's close to 50%?
The significance comes from cumulative compounding over many trades. Over 252 trading days per year, a 52% DA model makes 131 correct calls and 121 wrong ones — a +10 advantage. If each correct call earns a small positive expected return (say 0.05% per day on average), the 10-day edge translates to ~0.5% additional annual return. Compounded over a 14-year backtest (3520 days), this becomes statistically highly significant. The binomial test: P(≥131 correct out of 252 with p=0.5) ≈ 0.04, so at the 5% significance level, 52% DA is detectable. In professional quant trading, strategies with consistent 52–53% DA on liquid instruments are genuinely valuable — they form the basis of systematic equity market-neutral strategies.
Q12. Why does LSTM contribute value to the ensemble even with slightly worse MAE than XGBoost?
Ensemble theory shows that diversity among models is as important as individual accuracy. If MomentumAgent and SequenceAgent made identical errors (perfectly correlated), combining them would produce no benefit. But they learn different representations of the data — MomentumAgent uses lag features and technical indicators on tabular form; SequenceAgent uses raw return sequences through temporal gates. Their errors are partially uncorrelated. In practice, on days where momentum-based features give weak signals (e.g., near a trend reversal), the LSTM's sequence-based representation may still correctly capture the pattern. The Hedge algorithm will gradually increase SequenceAgent's weight during such periods. This is the core argument for using a heterogeneous ensemble — the whole is greater than the sum of its parts.
